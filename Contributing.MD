# Contributing to SweetEdition

Thank you for your interest in contributing to SweetEdition!  
This project is built to be modular, clean, and easy to extend.  
Whether you're fixing bugs, improving themes, or adding new modules — your help is appreciated.

---

## 🧭 How to Contribute

### 1. Report Issues
If you find a bug or visual inconsistency:
- Open an Issue on GitHub
- Include screenshots when possible
- Describe steps to reproduce the issue

### 2. Request Features
If you have an idea for a new module, theme variant, or UI enhancement:
- Open a Feature Request
- Explain the use case and expected behavior

### 3. Submit Pull Requests
To contribute code:
1. Fork the repository  
2. Create a new branch  
3. Make your changes  
4. Test thoroughly  
5. Submit a Pull Request with a clear description

---

## 🧪 Coding Guidelines

### JavaScript
- Use modern ES6+ syntax  
- Keep functions modular and readable  
- Avoid unnecessary dependencies  

### HTML / CSS
- Follow the SweetEdition neon‑purple aesthetic  
- Keep UI elements consistent with existing components  
- Use semantic HTML where possible  

### Shell Scripts
- Keep installer scripts POSIX‑compatible  
- Comment complex logic  
- Ensure scripts fail gracefully  

---

## 🔍 Testing Your Changes

Before submitting a PR:
- Test the update banner using a lowered `localVersion`
- Verify the web UI loads correctly on GitHub Pages
- Ensure installer scripts run without errors
- Confirm theme files apply correctly in KDE/Gtk environments

---

## 📦 Adding New Releases

When adding a new version:
1. Update `update.json`  
2. Add a new entry to `CHANGELOG.md`  
3. Upload the new tarball to the repo  
4. Tag the release in GitHub  

---

## 💬 Communication

If you want to discuss ideas before implementing them:
- Open a Discussion  
- Or start an Issue labeled **proposal**

Thanks again for helping make SweetEdition better!
